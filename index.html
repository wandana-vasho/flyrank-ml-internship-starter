00 — Abstract

Question: does a page's average search ranking position predict its click-through rate strongly enough to prioritize a content-review queue? Method: a hand-coded baseline rule and a grouped-validation Decision Tree / Logistic Regression pair were trained on 3.6M+ page-day rows from FlyRank's production search warehouse (month=2026-03), split by client to prevent identity leakage, and audited twice for label leakage. Result: the baseline scored higher on raw metrics (f1=0.994 vs. 0.818) — but only because it shared its formula with the label itself; once that coupling was named, the honest trained model (f1=0.818, precision=0.978) stood on its own using only position and volume, no same-day CTR required. The output is a three-tier, 735,691-row action playbook with explicit human-review rules and no automated changes.

01 — Question

The decision this work is for

A content team cannot manually review hundreds of thousands of pages. Something has to narrow the list — the question is whether that something should be a simple rule, or something learned.

Does a page's average ranking position meaningfully predict its click-through rate, and can that relationship be used to prioritize which pages a content team reviews first? This question traces across the full track: from an early correlation check on a small sample, through a full data contract on the real warehouse, to a validated model and a deployable playbook.

Decision supported Which pages a content/SEO team should prioritize reviewing or rewriting first, out of a page catalog too large to review manually.
02 — Data

What this is built on

FlyRank's internship-warehouse release (Hugging Face, gated), table fact_content_daily_performance, month 2026-03 — a mid-panel month, deliberately not the sealed final month reserved as a test period.

9,841,378total page-day rows, month=2026-03
3,611,061rows with usable GSC data (37%)
104pseudonymized clients in the release

Grain

One row = one page, on one day, for one client. Verified directly: total rows and unique (client, content, date) combinations matched exactly, with zero duplicates.

Filtering

Restricted to rows where gsc_data_available IS TRUE and gsc_impressions > 0. A minimum-impression floor of 50 was applied for the final action playbook, to exclude statistically unreliable low-volume rows from the priority queue.

Excluded, deliberately

  • fact_content_query_90d — a different grain (query-level), kept separate from this page-day analysis.
  • AI-referral columns and GA4-derived metrics — inconsistent coverage across clients, would bias toward GA4-enabled accounts only.
  • Raw client/content hash identifiers as model features — kept only as row identifiers, never as predictive inputs, to avoid the model memorizing specific clients.
Public-safe by construction All client and content identifiers arrive pre-hashed from FlyRank. No real client names, domains, URLs, or search queries appear anywhere in this paper or its underlying notebooks.
03 — Methodology

Label, features, and how the split was chosen

Label

underperforming = actual CTR below the expected CTR for the page's position tier, where expected CTR was measured directly from the warehouse: 0.0039 for positions 1–10, 0.0018 beyond position 10.

Features

Two, deliberately minimal: log_impressions (log-transformed impression volume) and avg_position (mean search ranking position). Both are same-day, point-in-time GSC metrics — available at the moment a decision would actually be made, with no dependency on future data.

Baseline

A hand-coded rule scoring each page by the same CTR-position gap used in the label, carrying a single reason code (CTR_BELOW_EXPECTED_FOR_POSITION) and a binary action label.

Model

A Decision Tree Classifier (max depth 4, balanced class weights), compared against Logistic Regression as a calibration check.

Validation design

Grouped by client (GroupShuffleSplit): 37 clients in training, 10 held out for test, zero client overlap confirmed directly. This was chosen specifically because per-client data history varies widely in this release — a random row-level split risks letting the model memorize individual clients rather than learn a pattern that generalizes.

Leakage checks

Run twice. First, a feature-vector audit tested the final feature set against label-derived and circular columns. Second, a follow-up validation pass deliberately re-added the label-derived column to confirm the size of the resulting inflation: +0.182 f1, before removing it and keeping only the honest, leak-free result.

04 — Results

The honest table

The baseline wins on paper. That is the finding worth explaining, not hiding.

MethodPrecisionRecallF1
Baseline (rule-based)1.0000.9890.994

Logistic Regression (grouped split)0.9790.6990.816
Why the baseline "wins" The baseline score and the label were built from the same underlying comparison — CTR against a position-tier expectation — at slightly different thresholds. It is closer to restating the label than predicting it. A fair benchmark would use a fixed rule that does not share its formula with the label.

What the trained model actually offers

Both models reach ~0.98 precision using only log_impressions and avg_position — without ever seeing same-day CTR. That distinction matters whenever CTR for the decision period is not yet known, which the baseline's formula quietly assumes away.

Feature importance

log_impressions accounted for 96.2% of the Decision Tree's splits; avg_position only 3.8% — despite position being the more theoretically motivated signal throughout this work. The likely explanation: low-impression pages produce noisier, more extreme CTR values by construction (a single click on 3 impressions swings CTR wildly), so the model may be leaning on volume as a proxy for measurement reliability rather than position capturing the true effect directly.

A validation check, reported honestly

Re-running the same model under a naive, non-grouped random split produced f1=0.836 — only 0.018 higher than the grouped-by-client result (f1=0.818). Grouped validation remains the methodologically correct choice, but the gap here was smaller than expected, likely because the two features used are generic page-level signals rather than client-specific ones.

05 — Limitations

What this cannot claim

  1. Single-month snapshot. Built on 2026-03 only; seasonal or campaign-driven CTR shifts are not captured.
  2. Baseline-label coupling. As shown in Results, the baseline is not an independent benchmark for this task.
  3. Unbalanced client history. Clients in this release have very different data start dates; any pattern found may be shaped disproportionately by longer-history clients.
  4. Client concentration in early findings. An earlier top-20 review found 15 of 20 highest-priority rows belonged to a single client — plausibly real, plausibly a client-specific tracking anomaly. Not resolved here.
  5. Observational, not causal. Position and CTR are measured together, not manipulated. Title quality, brand recognition, and query intent likely contribute alongside position.

What should not be automated

  • No automatic title/meta rewriting — every change needs a human draft.
  • No automatic publishing or client-facing alerts without review first.
  • No bulk action on a client-concentrated cluster without checking for a tracking anomaly.

Monitoring

Re-run monthly, not more often — CTR needs volume to stabilize. If recomputed position-tier benchmarks drift more than ~20% from the values used here, thresholds should be recalculated rather than reused as-is.
